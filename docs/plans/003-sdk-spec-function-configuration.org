* SDK Spec Function Configuration

Source spec: https://github.com/inngest/inngest/blob/main/docs/SDK_SPEC.md

** Goal

Expand function configuration support so registration payloads can represent the
function-level controls documented in the spec while preserving the current Elixir
SDK API where practical.

** Implementation Guide

Before checking off any implementation item in any phase, decide any relevant
open question, capture the current behavior if it matters, add or update a
focused test that fails for the desired spec behavior, implement the smallest
change, and leave the targeted test green. Do not end a phase with intentionally
failing tests.

** Decisions

- Public function config remains idiomatic Elixir ~snake_case~. Spec ~camelCase~ names are emitted only at the registration JSON boundary.
- Invalid function config raises during serve/registration payload generation, matching existing validation behavior. Module loading should remain side-effect-light.
- ~batch_events.key~ does not change conflict semantics. Any non-nil ~batch_events~ config remains incompatible with ~rate_limit~ and ~cancel_on~.
- New function config options use raw maps with atom keys, matching existing options. Do not add dedicated structs or keyword-list normalization in this plan.
- ~singleton.mode~ uses atom values, ~:skip~ or ~:cancel~, in the public Elixir API and renders the corresponding spec string in registration payloads.
- Update function configuration docs/examples to cover the full supported option set after this plan, including existing and new options, while avoiding unrelated documentation rewrites.
