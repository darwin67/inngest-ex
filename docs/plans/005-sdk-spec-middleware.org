* SDK Spec Middleware

Source spec: https://github.com/inngest/inngest/blob/main/docs/SDK_SPEC.md

** Goal

Add the required middleware API, lifecycle hooks, mutation points, and
capability boundaries required for SDK spec compliance.

** Implementation Guide

Before checking off any implementation item in any phase, decide any relevant
open question, capture the current behavior if it matters, add or update a
focused test that fails for the desired spec behavior, implement the smallest
change, and leave the targeted test green. Do not end a phase with intentionally
failing tests.

** Decisions

- Middleware will be represented by behaviour modules with optional callbacks.
  Middleware configuration accepts modules and ={module, opts}= tuples so hooks
  remain named, discoverable, and testable.
- Public middleware callbacks follow the current TypeScript SDK hook model:
  ~on_register~, ~transform_*~, ~wrap_*~, and ~on_*~ lifecycle hooks. Elixir uses
  behaviour modules and ={module, opts}= entries instead of TypeScript classes.
- Middleware can be configured on first-class clients and per function through
  function options. Global application middleware may remain as a compatibility
  fallback, but the primary public API should be client-owned.
- Client-level middleware attaches to ~use Inngest.Client~ options and is
  resolved onto the runtime ~%Inngest.Client{}~ used by routers, sends,
  introspection, and call execution.
- Event-send middleware applies to both ~Inngest.Client.send/2~ and
  ~Inngest.StepTool.send_event/3~ through the same event-send hook path.
  ~StepTool.send_event/3~ already receives the active invocation client through
  ~Inngest.Function.Context~, so middleware must preserve that client scope.
  Durable step memoization stores the result after middleware and event send
  complete.
- Connect, checkpointing, streaming, AI Gateway, Gateway fetch, and in-band sync
  remain deferred in this plan. Plan 5 only documents those boundaries and
  prevents optional capabilities from being advertised before implementation.
- Introspection must only advertise ~trust_probe~ after a separate trust probe
  feature exists. Signed introspection alone is not sufficient to advertise that
  capability.

** Phase 2: Mutation And Capability Boundaries

- [X] Allow middleware to add fields to function context.
- [X] Allow middleware to access framework/platform request information where available.
- [X] Allow middleware to mutate memoized step data before user code sees it.
- [X] Allow middleware to mutate output and errors before SDK responses are sent.
- [X] Allow middleware to mutate events before sending them to Inngest.
- [X] Keep Connect, streaming, checkpointing, AI Gateway, Gateway fetch, and in-band sync as explicit future work unless implemented in this phase.
- [X] Ensure introspection capabilities only advertise implemented optional features.
